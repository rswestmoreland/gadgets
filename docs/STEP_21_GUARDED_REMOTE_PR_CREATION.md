# Step 21 - Guarded Remote PR Creation

Status: implemented at checkpoint/code level.

## Goal

Step 21 adds a narrow remote pull request creation path for the Developer Pack.
It is intentionally not a general GitHub/GitLab automation layer and it does not
push branches.

Implemented command:

```bash
gadgets git pr create [--project <path>] <approval-request-id> --body-run <run-id> --head <branch> [--base <branch>] [--title <title>]
```

## Configuration gate

Remote PR creation is disabled by default. A project must explicitly opt in with
`.gadgets/config.yaml`:

```yaml
git:
  remote_pr:
    enabled: true
    provider: github
    owner: example-owner
    repo: example-repo
    api_base: https://api.github.com
    token_env: GITHUB_TOKEN
    default_base_branch: main
```

Only `provider: github` is implemented in this checkpoint. The token value is
read from the configured environment variable and must not be written to
evidence.

## Required gates

The remote PR provider requires all of the following before making the API call:

- `git.remote_pr.enabled: true`
- configured GitHub owner and repo
- configured token environment variable
- valid head and base branch names
- head and base branch must differ
- verified patch approval request
- approved approval record
- unexpired approval if `expires_at` is present
- approval scope hash must match request scope hash
- completed local PR body evidence bundle
- deterministic policy approval for `git.pr.create`

## Runtime behavior

The provider creates one GitHub pull request through the GitHub API. It uses the
local PR title/body evidence generated by Step 19 and the explicit head/base
branches supplied to the CLI.

It does not push branches. The head branch must already exist remotely.

## Explicit non-goals

Step 21 does not implement:

- Git push
- Git pull
- Git fetch
- Git merge
- Git rebase
- checkout or switch
- arbitrary shell
- arbitrary Git arguments
- model-provider tool execution
- patch application
- test execution
- Linux admin behavior
- database/cloud/deployment behavior
- GitLab support
- fork-style `owner:branch` head refs

## Evidence

Remote PR creation writes a separate evidence bundle with:

- `summary.md`
- `bundle.yaml`
- `remote_pr_request.txt`
- `approval_verification.txt`
- `pr_body_reference.txt`
- `http_status.txt`
- `remote_pr_response.txt`
- `remote_pr_url.txt`
- `duration.txt`
- `policy_decision.txt`
- `assumptions.txt`

The remote API response is capped and redacted for obvious secret-like lines
before evidence write.

## Audit events

The provider appends audit events including:

- `git.pr.create.requested`
- `approval.verification_failed` when applicable
- `policy.checked`
- `action.allowed` or `action.denied`
- `git.pr.create.started`
- `git.pr.create.completed` or `git.pr.create.failed`
- `evidence.created`
- `run.completed`

## Safety notes

Remote PR creation is the first implemented external Git provider action. It is
therefore explicit, disabled by default, approval-bound, evidence-bound, and
config-bound. It should be validated externally before broader remote Git or PR
work is added.
